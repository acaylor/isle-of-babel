# Architecture

How Isle of Babel is put together. The short version: **three scenes, one
autoload, and a procedural toolkit** — every mesh, material, book, and sound
of place in the game is generated by code at runtime, which is why the
repository contains no binary assets at all.

For the philosophy behind these choices, see [VISION.md](VISION.md). For how
to work in the codebase, see [CONTRIBUTING.md](CONTRIBUTING.md).

## The one-node scene pattern

Every `.tscn` in this repo is ~6 lines: a single `Node3D` with a script
attached. The script's `_ready()` builds the entire world — terrain, water,
lighting, props, colliders, the player — from code. There is no scene-tree
authoring in the Godot editor; the editor is optional for development.

This is deliberate. It keeps the repository 100% reviewable text, makes every
change a readable diff, and forces all generation to be deterministic and
reproducible.

```
scenes/island.tscn   →  [Node3D "Island"  + island.gd]   builds everything in _ready()
scenes/library.tscn  →  [Node3D "Library" + library.gd]
scenes/forest.tscn   →  [Node3D "Forest"  + forest.gd]
```

## Scene flow and the Game autoload

`autoload/game.gd` is the only autoload (registered as `Game`). It owns:

- **The input map** — all actions (WASD, jump, sprint, interact `E`, flip
  `F`, summon `Q`) are registered programmatically at startup, so
  `project.godot` stays minimal.
- **Scene travel** — `Game.travel(scene_path, spawn)` fades to black, swaps
  the scene, and fades back in. `Game.blink(action)` is the quick in-scene
  variant (the crystal, the wizard's last door).
- **Spawn-point routing** — `Game.spawn_point` is a string the *destination*
  scene reads in its own `_spawn_player()`. Each scene keeps a dictionary of
  named spawn transforms and handles a few special sentinels.
- **The pause menu** and the game `VERSION` constant shown on the title
  screen.

The full routing graph:

```
                        "menu" (sentinel: island builds the title screen, no player)
                          │
                 ┌────────▼─────────┐
    "dock" ─────▶│      ISLAND      │◀──── "tower_door"   (library arch door)
   "voyage" ────▶│  island.gd       │◀──── "tower_top"    (library dais portal / crystal)
 (boat arrival)  └───┬─────────┬────┘
                     │         │
        tower door   │         │  E on the rowboat
     "entrance" ─────▼──┐   ┌──▼────── "voyage" (boat arrival leg)
              ┌─────────┴┐  ├──────────────────┐
              │ LIBRARY  │  │      FOREST      │◀── "jetty"
              │library.gd│  │    forest.gd     │
              └──────────┘  └──────────────────┘
      "portal" (balcony portal)      E on the boat → back to ISLAND "voyage"
```

Two sentinels get special handling: `"menu"` makes the island build the
orbiting-camera title screen instead of spawning a player, and `"voyage"`
makes the island/forest spawn the player *seated in the moving boat* for the
arrival leg of a crossing.

## The player

`player/player.gd` (`class_name Player`) is a self-building first-person
controller: its `_ready()` constructs its own capsule collider, camera,
interaction `RayCast3D`, and HUD (crosshair, prompt label, message panel, and
the full open-book reading UI). Scenes just instantiate the script.

Notable mechanics, each of which encodes a lesson learned:

- **Interaction** — `_update_interact()` raycasts from the camera every
  physics tick; if the hit is an `Interactable`, it shows `[E] <prompt>` and
  fires `Interactable.interact()` on a press. The `_consume_interact` flag
  exists because input events are processed *before* physics in the same
  frame: the E press that closes a book would otherwise still register as
  `just_pressed` when `_update_interact()` runs, immediately taking a new
  book off the shelf.
- **Reading mode** (`_reading`) — movement damps to zero, E/Esc close, F
  leafs (randomly for shelf books, in order for fixed texts like the
  journal, via `_book_pages`).
- **Passenger mode** (`_seat`) — the player's position rides a seat node
  (the boat) while the head stays free to look around. Physics, gravity, and
  interaction are suspended while seated.
- **Terrain feel** — `floor_max_angle` is raised to 55° (Godot defaults to
  45°) and `floor_snap_length` to 0.35, because noisy procedural terrain
  constantly bounces and blocks a default controller.

## The Interactable pattern

`scripts/interactable.gd` is the whole interaction system: a `StaticBody3D`
with a `prompt` string and an `action: Callable`.

```gdscript
root.add_child(Interactable.make(shape, "Read the boundary stone",
    func(p: Node) -> void: (p as Player).open_book(BookLore.tablet()),
    Transform3D(Basis.IDENTITY, Vector3(0, 1.1, 0))))
```

`prompt` and `action` are plain vars, so behavior can be swapped at runtime —
that's how the ruin's cold ring becomes "the wizard's last door" when the
fifth kept book is returned.

## The procedural toolkit

### Forge — primitives (`scripts/forge.gd`)

Static helpers for the boring parts: `box`, `cyl`, `sphere`, `torus`,
`mesh`, colliders, omni lights, and the reusable glowing `portal`. Two
materials matter:

- `Forge.mat(color, rough, emission, energy)` — plain `StandardMaterial3D`.
- `Forge.vc_mat(rough)` — vertex-color material with
  **`vertex_color_is_srgb = true`**. All hand-authored vertex colors in this
  project are sRGB; without that flag Godot treats them as linear and the
  whole world renders washed-out and pale. This bit us once; it lives in one
  place now.

### Flora — nature meshes (`scripts/flora.gd`)

Detailed nature meshes as `SurfaceTool` triangle soup: pines, broadleafs
(with an autumn variant), bushes, boulders (with a moss parameter), grass
tufts and wildflowers, ferns, mossy fallen logs with mushrooms, and unique
ridged mountains with forested slopes and snow caps.

The load-bearing convention: **normals are always computed by hand** (radial
for tubes, sphere-direction for blobs, grid-cross for mountain surfaces,
straight `UP` for grass and ferns so lawns light evenly), and every material
uses `cull_mode = CULL_DISABLED`. Together these make lighting completely
independent of triangle winding order — generation code never has to worry
about which way a triangle faces, which removes an entire class of
invisible-face bugs.

### Determinism

Every generator seeds its own `RandomNumberGenerator` with a fixed constant
(`rng.seed = 1337`, `4242`, …). The same build of the game always produces
the same island, the same forest, the same book placement. The library goes
further: each cell derives its seed from its grid coordinates
(`c.x * 73856093 + c.y * 19349663`), so any cell is reproducible in
isolation, no matter what order cells stream in.

## Terrain: `height_at()` is the single source of truth

Each outdoor scene defines one function, `height_at(x, z) -> float`, and
*everything* consults it: the terrain mesh, the trimesh collider, vegetation
placement, path stones, water lines, spawn heights. Terrain features are
composed as additive/lerped terms inside it:

- island: radial island shape, tower plateau (with a gentle south **ramp**
  so the path climbs to the door instead of hitting a lip), the dock
  corridor, noise detail
- forest: coast gradient, inland rise, the stream channel (which begins at
  the spring, not the map edge), the ruin clearing plateau, **boundary
  cliffs** — meandering ~70° rock walls past the player's 55° slope limit,
  so the world ends in stone and water with no invisible walls

Two supporting patterns:

- **Landmark exclusion** — `_clear_of_landmarks(x, z)` keeps scattered
  nature off trails, clearings, secrets, and the scree apron below cliffs.
- **Plan before terrain** — anything the terrain must *meet* (the
  footbridge's earthen abutments) is planned in `_ready()` *before* the
  terrain mesh is built, so `height_at()` can raise the ground flush with
  it. Order matters: geometry that ignores this ends up hovering.

The collider is `mesh.create_trimesh_shape()` with
`backface_collision = true` — again, winding-independence.

## The infinite library

`scenes/library.gd` streams 14 m gallery cells around the player:

- `_stream_cells()` runs every frame; wanted cells within `LOAD_RADIUS` (2)
  are built at most `BUILDS_PER_FRAME` (2) per frame, sorted nearest-first;
  cells beyond `UNLOAD_RADIUS` (4) are freed.
- Each cell owns its **west and north edges plus one corner column**, so the
  infinite lattice has no duplicated geometry where cells meet.
- Books are `MultiMesh` instances (hundreds per shelf) with per-instance
  color; shelves share one `ArrayMesh`.
- Cell (0,0) is the entrance chamber (door back to the island, dais portal
  to the summit); every other cell furnishes itself from its deterministic
  RNG — reading tables, book piles, rugs, lamps.

The creative way home is the **paper guide bird** (`Q`): a page folds itself
into a glowing bird that flies toward the entrance at shelf-top height,
waiting when the reader falls behind, dissolving over the dais.

## Books and lore

`scripts/book_lore.gd` has two halves:

- **Generated books** (`random_book()`): a subject is picked once and
  threaded through the title, chapter heading, and eight body sentences, so
  every shelf book reads as one train of thought instead of madlibs.
- **Fixed texts**: the wizard's four-entry journal, the boundary stone, the
  five kept books (each a distinct annotated voice), and the stranger's
  note. The written texts carry the actual story; the generated ones carry
  the atmosphere — and they cross-reference each other ("what the forest
  reads, it keeps" in the journal is the kept books' premise).

## Ambient animation

Scenes keep tiny FX registries — pulsing emissive materials and lights
(each on its own phase), slowly turning orbit pivots (pages circling a kept
book), and drifting motes (sine-bobbed) — all updated in a single `_process`
pass. Cheap, allocation-free, and enough to make the world feel alive.

## Testing

Two headless tools, documented in the README's Testing section:

- **The smoke test** (`tests/smoke.tscn` + `observer.gd`) plays the whole
  game in ~55 seconds of simulated time: island → library (opens a book via
  *real simulated E key events* through `Input.parse_input_event`, exercising
  the same input path a player uses) → summit → title screen → forest
  (validates every fixed text, collects all five kept books and asserts the
  ring wakes, **walks** the footbridge and the boundary cliffs and the tower
  ramp with held movement input) → rides the boat's arrival leg to the
  jetty. Stages are time-based because scene fades are.
- **The capture harness** (`tests/capture.tscn`) is an env-driven camera/
  spawn rig for Godot's movie-maker mode — every visual change in this
  project is verified by rendering real frames and looking at them.

House rule: when a test is added for a bug, the old broken code is
temporarily restored to prove the test fails (see commit messages quoting
their negative repros — "walked up and over the boundary cliffs, x=152.9").

## Release flow

Feature branch → Gitea PR → merge is the release point → stamp the
CHANGELOG date on `main` → tag `vX.Y.Z-alpha` as a pre-release.
`Game.VERSION` is bumped inside the feature PR so the title screen always
matches the tag.
